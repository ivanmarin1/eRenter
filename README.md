# Booking Calendar — KMP + Compose Multiplatform

A modular, clean-architecture vacation-rental calendar. The **calendar** is a self-contained,
reusable feature designed to drop into a larger booking app. It shows, per apartment, when a
guest is **checking out** and when the next one is **checking in**, so turnover / cleaning days
are obvious at a glance.

## Module graph

```
composeApp                       (Android + iOS + Desktop host; composition root / DI wiring)
  └─ feature:calendar:ui         Compose UI (stateless screen + components). REPLACEABLE.
       └─ feature:calendar:presentation   ViewModel + UiState + intents. No Compose.
            └─ feature:calendar:domain    Entities, use cases, repository interface. Pure Kotlin.
                 ▲
                 └─ feature:calendar:data Repository impl + swappable data source (+ Koin).
  └─ core:designsystem           Theme + calendar colour tokens. Shared across features.
```

Dependency rule (clean architecture): arrows point **inward**. `domain` depends on nothing
framework-specific. `data` implements `domain` interfaces. `presentation` depends only on
`domain`. `ui` depends only on `presentation` + `designsystem`. `data` and `ui` never see each other.

### Why it maps to your requirements
- **Reusable across projects** — `feature:calendar:*` is a vertical slice with no app-specific
  code. Another project includes the four modules and drops in `CalendarRoute()`.
- **UI decoupled / replaceable** — all business + date logic lives in `domain`/`presentation`.
  The `ui` module only renders a `CalendarUiState` and emits `CalendarEvent`s. Delete it and
  write a new UI without touching anything else.
- **Uncluttered, recognisable check-in/out** — each day cell is split on the diagonal:
  top-left = check-out (amber), bottom-right = check-in (green). Both filled + a red dot = a
  **turnover** (clean between guests). Guest/apartment names appear only when you tap a day.
- **Fits a larger booking app** — the domain `BookingRepository` is the seam. Swap the
  in-memory source for SQLDelight or a backend, or have the bigger app implement the interface.

## Running
Open the folder in **Android Studio** (with the Kotlin Multiplatform plugin). Let it sync,
then use the generated run configurations:
- **Android**: run `composeApp` on a device/emulator.
- **Desktop**: `./gradlew :composeApp:run`
- **iOS**: run the `iosApp` configuration (Android Studio wires the Xcode project & framework).
- **Tests**: `./gradlew :feature:calendar:domain:allTests`

> The Gradle **wrapper JAR** is generated by the IDE (or `gradle wrapper`) on first open.
> Copy `local.properties.sample` → `local.properties` and set your Android SDK path.

## Where to extend next
- Add a booking-entry feature module (`feature:booking-entry:*`) to replace the paper notes;
  it reuses the same `BookingRepository`.
- Replace `InMemoryBookingDataSource` with SQLDelight for offline persistence.
- Extract the per-module Gradle boilerplate into `build-logic` convention plugins.

## Versions
Kotlin 2.1.0 · Compose Multiplatform 1.7.3 · AGP 8.5.2 · Koin 4.0.0 · kotlinx-datetime 0.6.1.
If your IDE bundles different versions, adjust `gradle/libs.versions.toml` and re-sync.
